# Design Doc: volunteer-recommendation-refocus

本设计用于把当前项目从“人生经济模型/路径推演”纠偏为“高考志愿与专业选择辅助系统”。实现由其他 agent 负责，本文件作为设计方向、实现边界和验收依据。

## 1. 背景与问题

项目原目标是给高考生和家长做志愿填报参考，以及较长期的职业规划辅助。当前实现中出现了两个需要纠正的方向：

- “三条人生路径 / 人生轨迹”会暗示孩子成长路径被系统固化，不适合出现在高考志愿辅助产品中。
- “回本周期 / ROI / 15年净收益”等表达把上大学描述成交易，不适合面向孩子和家长的志愿决策场景。

费用、就业、薪资区间仍然有价值，但必须作为“约束与参考”呈现，而不是“投资回本”。

## 2. 产品定位

产品定位调整为：

> 高考志愿与专业选择辅助系统：基于政策规则、考生分数与单科能力、一分一段、历年录取数据、家庭预算和院校专业组信息，生成可解释的专业方向建议与冲稳保院校清单。

产品不输出：

- 人生路径、人生轨迹、人生赛道、命运路线等固化叙事。
- 回本周期、投资回报率、ROI、15年净收益等交易化叙事。
- “保证录取”“保证就业”“未来收入预测”等确定性承诺。

产品可以输出：

- 大学期间总开销，包括学费、住宿费、城市生活费。
- 家庭预算压力，包括可承受、明显负担、超预算等。
- 就业参考，包括起薪三分位、就业密度、岗位城市分布、专业学习风险。
- 录取风险，包括冲、稳、保、偏冲、偏保，以及数据粒度说明。

## 3. 推荐主流程

推荐链路应从“专业方向优先”出发，而不是从“人生路径”出发。

```
考生画像
  -> 政策与资格过滤
  -> 位次与历年同制度数据对照
  -> 专业方向适配评分
  -> 院校专业组/学校候选生成
  -> 费用与家庭承受压力评估
  -> 冲稳保志愿建议
  -> 可解释输出
```

### 3.1 考生画像输入

必须支持以下字段：

- 省份、年份、总分。
- 首选科目：物理/历史。
- 再选科目：3+1+2 中的两门再选科目。
- 数学分数。
- 外语语种与外语分数。
- 实际英语能力，用于区分“日语高考分数”和“英语学习适配”。
- 家庭预算：家庭年收入、可投入储蓄、每年教育预算、可接受助学贷款、是否接受民办/中外合作。
- 偏好字段可选：城市偏好、专业兴趣、是否倾向公办、是否接受考研依赖强专业。

### 3.2 政策与资格过滤

资格过滤是硬约束，必须先于评分：

- 选科要求不满足的专业组不得进入推荐。
- 高考外语语种被章程限制时，不满足则不得进入推荐。
- 单科门槛由章程明确时才作为硬过滤。
- 数学弱、英语弱但章程未硬性限制时，只标记学习风险，不擅自剔除。
- 必须保留不可报原因，供前端展示。

### 3.3 位次与录取判断

推荐不得只看分数，必须看位次：

- 使用当年一分一段表将总分转换为当年位次。
- 若使用历年录取数据，优先使用同制度数据。例如河南 2025 起进入新高考，应优先用 2025 及以后同制度数据。
- 跨年比较要显式标注数据年份与制度差异。
- 录取判断区分“专业组投档可能性”和“组内目标专业风险”，不得把“可投进专业组”写成“可录取目标专业”。

### 3.4 专业方向评分

专业推荐应拆成两个分数：

- `market_value`：专业方向外部市场参考，来自岗位密度、薪资三分位、城市覆盖、长期趋势。
- `student_fit`：考生适配，来自选科资格、数学学习风险、英语适配、兴趣与学习门槛。

综合专业价值采用乘法门控：

```
major_value = market_value * student_fit
```

这样可以避免“市场热但孩子明显学不动”的专业被强推。

### 3.5 院校与专业组候选

每个推荐项必须说明数据粒度：

- `major`：有目标专业录取线，可信度最高。
- `group`：只有院校专业组投档线，必须提示组内专业不确定。
- `school`：只有学校级最低投档线，只能作为候选，不得声称目标专业可录。

院校排序应综合：

- 专业方向适配。
- 录取档位。
- 院校性质：公办、民办、中外合作。
- 费用压力。
- 城市就业参考。
- 学校平台与专业资源。
- 数据可信度。

### 3.6 费用压力

费用模块保留，但只做“大学期间开销与家庭承受能力”：

- 展示 4 年学费、住宿费、生活费、总开销。
- 区分公办、民办、中外合作。
- 给出家庭压力等级：可承受、有压力、明显负担、超预算。
- 不计算回本周期，不计算教育 ROI，不展示多年净收益。

## 4. API 与模型调整方向

现有 `/volunteer/recommend` 可以保留为传统冲稳保接口。

建议新增或替换主接口：

```
POST /api/v1/volunteer/advisory
```

语义：输入完整考生画像，输出专业方向建议和院校专业组清单。

建议逐步废弃或隐藏：

```
POST /api/v1/volunteer/life-paths
POST /api/v1/volunteer/life-trajectory
```

如果短期保留兼容，应改为内部 deprecated，不在 UI、README、主导航中出现。

### 4.1 输出结构建议

```python
VolunteerAdvisoryResult:
  student_rank: int
  province: str
  track: str

```

## 5. 前端调整方向

导航和页面应调整为：

- `专业推荐`：主入口。
- `志愿表`：冲稳保院校清单。
- `大学费用`：费用估算。
- `位次工具`：分数/位次双向查询。
- `省控线`：政策基础数据。

必须移除或替换：

- “人生路径”导航。
- “人生经济模型模拟器”副标题。
- “填报志愿，看见孩子的人生轨迹”。
- “回本周期”卡片。
- “ROI / 15年净收益 / 投资回报”相关字段。

建议主标题：

> 高考志愿专业推荐

建议副标题：

> 基于分数、选科、外语、数学、一分一段和家庭预算，生成可解释的专业方向与冲稳保院校建议。

## 6. 实现验收标准

实现完成后按以下标准验收。

### 6.1 产品口径验收

全仓库搜索不得在用户可见页面、README 主说明、API 描述中出现以下词作为功能名称或推荐结果：

- 人生路径
- 人生轨迹
- 回本
- ROI
- 投资回报
- 15年净收益
- 命运
- 赛道

允许在归档文档或 deprecated 注释中出现，但必须明确为历史遗留或废弃接口。

### 6.2 推荐逻辑验收

- 输入日语考生时，英语语种限定专业必须被过滤或明确标记不可报。
- 输入数学较弱考生时，数学强依赖专业不得仅因市场热度排到前列。
- 选科不满足时，专业组不得进入可推荐结果。
- 只有专业组数据时，结果必须提示“组内目标专业数据不足”。
- 民办/中外合作高费用选项必须显示费用压力，不得只按学校或专业分排序。
- 输出至少包含专业方向建议、冲稳保学校清单、费用压力、不可报原因、数据来源说明。

### 6.3 费用模块验收

- 展示大学期间总开销，而不是回本。
- 费用拆分包含学费、住宿费、生活费。
- 公办、民办、中外合作费用差异在 UI 中可见。
- 家庭预算不足时，结果标记超预算或明显负担。

### 6.4 数据与政策验收

- 使用一分一段表完成分数到位次转换。
- 当前年份数据缺失时允许回退，但必须在 `source_note` 或 `notes` 中说明回退年份。
- 2025 及以后河南新高考数据不得与旧文理科制度简单平均。
- 政策规则、招生章程、专业组数据不足时不得伪装成确定结果。

### 6.5 测试验收

实现方至少补充或更新以下测试：

- API 测试：新主接口返回专业方向、学校清单、费用压力、不可报原因。
- Engine 测试：专业方向评分使用 `market_value * student_fit`。
- Eligibility 测试：外语、选科、单科门槛分别覆盖。
- UI 文案测试或静态扫描：禁止词不出现在用户可见主流程。
- 回归测试：位次查询、大学费用、原冲稳保接口仍可用。

## 7. 审核方式

设计审核按三层进行：

1. 产品审核：是否彻底去掉人生固化和交易化表达。
2. 算法审核：是否先资格过滤，再专业适配，再院校匹配，再费用压力。
3. 结果审核：是否能让家长看懂“为什么推荐/为什么不可报/家庭是否扛得住”。

若实现只改文案但保留回本模型作为主流程输出，不通过验收。

若实现只按学校分数冲稳保推荐，未体现专业方向、选科、外语、数学、费用压力，不通过验收。

若实现把专业组投档误写成目标专业录取，不通过验收。

## 8. 后续可单独立项

以下内容不进入本次重构范围：

- 自动爬取最新招生章程。
- 招聘平台岗位数据 ETL。
- 城市租房数据 ETL。
- 全国全量院校专业组数据补齐。
- 多年职业收入曲线模拟。

这些能力可以后续作为数据工程或职业规划模块单独设计，但不得重新引入“回本”作为志愿填报主指标。
