## Context

承接已归档的 data-foundation（6 实体 + SourcedRecord + Loader 架构，94 测试）与 provincial-score-data（省控线 + 一分一段表 + 位次法双向查询，133 测试）。本 change 是三层架构第②层 decision-engine 的志愿筛选子能力——把"考生分数 → 可报学校 → 冲稳保志愿表"这条决策链路打通。

位次法已就绪（rank_query.score_to_rank/rank_to_score），但仅是"分数↔位次"的换算工具，缺上层"用位次筛学校、分冲稳保、生成表"的引擎逻辑。本 change 填补这一层。

## Goals / Non-Goals

**Goals:**
- 志愿填报引擎纯函数：位次换算(含缺失插值) / 等效位次折算 / 冲稳保分档 / 志愿表生成 / 录取概率估算
- 录取数据 schema 扩展支持新高考（院校专业组 + 选科要求 + 本科批合并批次）
- API 端点：recommend（生成志愿表）+ admissions（按分数范围筛选可报学校）
- 真实录取种子数据（河南核心校，覆盖冲到保梯度）
- 复用 data-foundation 纯函数引擎模式（无 DB 依赖，router 取数喂 domain 对象）

**Non-Goals:**
- ❌ 批量录取数据（30-40 校 OCR）→ 阶段 B 后续扩充（现 6 校足够验证引擎）
- ❌ 2026 投档线（未公布，用 2024 历年位次预测）
- ❌ Web 前端 → change③ web-ui
- ❌ 人生路径推演（收入/城市生存/职业模拟）→ change② decision-engine，与本 change 解耦
- ❌ 爬虫实时抓取 → 后续可选 change

## Decisions

### D1: 引擎层保持纯函数，无 DB 依赖（复用既有模式）

**选择**：`volunteer.py` 全部顶层纯函数，输入 domain 对象（StudentProfile/AdmissionRecord/ScoreRankEntry），输出 VolunteerTable。DB 取数由 router 完成。

**理由**：与 insurance.py/rank_query.py 一致，纯函数易测试、可组合、无副作用。引擎只做决策逻辑，不碰持久化。

### D2: 冲稳保阈值用相对位次比，非绝对分数差

**选择**：`ratio = school_rank / student_rank`。ratio<0.95→冲（学校更难录），0.95-1.05→稳，>1.05→保。冲的志愿设 SPRINT_FLOOR=0.40（学校位次低于考生 40% 则不列入，几乎录不到）。

**理由**：位次比是位次法的天然度量，不受试卷难度/总分波动影响。绝对分数差在跨年/跨省时不可比。±5% 稳定带是志愿填报经验值（位次接近=录取概率高）。

**替代方案**：绝对分数差（跨年不可比，否）、固定位次区间（不随考生水平自适应，否）。

### D3: 等效位次用考生总量比例折算（近似，标注置信度）

**选择**：`历史等效位次 = 今年位次 × (历史考生总量 / 今年考生总量)`。无历史数据时返回原位次（降级不阻断）。

**理由**：解决"今年 35 万 vs 去年 40 万考生"位次不可直接比。比例折算是行业常用近似法（精确法需本科上线人数比例，数据更难获取）。结果在 VolunteerTable 标注等效位次，供前端展示。

### D4: 风险偏好驱动各档配额（StudentProfile.risk_preference 复用）

**选择**：冲/中/稳 三档对应不同 sprint/stable/safe 配额（如稳型=2冲3稳5保，冲型=6冲3稳2保）。各档内按录取概率降序裁剪。

**理由**：StudentProfile 已有 risk_preference 枚举（稳/中/冲），直接驱动志愿表结构，符合"稳型考生求保底、冲型考生敢搏"的真实需求。无需新增输入字段。

### D5: 录取概率用位次比的线性模型（粗估，标注 basis）

**选择**：`probability` 基于位次比 ratio：ratio=1→0.9，ratio 越小（越难）概率越低（线性衰减），ratio 越大（越容易）趋近 0.99。每条建议带 `basis` 说明（如"位次比 0.46"）。

**理由**：真实录取概率需多年数据建模（如 logistic 回归），现仅 1 年数据不足以训练。线性粗估够前端展示"大概难度"，basis 字段透明化估算依据，不误导用户。

## Risks / Trade-offs

- **[录取数据覆盖少]** 现 6 校，筛选结果有限 → 缓解：阶段 B OCR 扩充；引擎已支持任意数量数据，扩充无需改代码
- **[等效位次近似]** 比例折算非精确 → 缓解：D3 标注置信度，前端展示原始/等效位次对比
- **[概率模型粗]** 线性估算不准 → 缓解：D5 basis 字段透明化，未来多年数据可升级 logistic
- **[一分一段表年份不齐]** API 用 2024 录取数据但 2024 一分一段表不全（CSV 到 2024 有，2026 有，中间 2025 缺）→ 缓解：引擎插值容错；前端标注数据年份
- **[新高考选科匹配]** 专业组选科要求需与学生选科对齐 → 缓解：schema 已加字段，引擎目前仅透传展示，严格匹配留待数据齐备后增强

## Open Questions

1. 是否需要"专业级"志愿表（现按学校最低投档分，未到专业组/专业粒度）？建议：阶段 B 拿到专业组数据后再增强，现阶段学校级够用。
2. 录取概率是否需引入"院校大小年"（录取位次年度波动）修正？建议：需 3 年数据，待阶段 B 补齐后考虑。
